# Client Integration Contract

This document defines the relationship between Nexora client apps and the core platform.

## Client Apps

Initial client:

- Android TV / Fire TV

Future clients:

- Android Mobile
- iOS
- Apple TV
- Samsung Tizen
- LG webOS
- Web Player

## Core Principle

Clients are player/access clients.

The platform is account/device/license/config authority.

The platform is not content/provider/stream authority.

## Device Identity

Primary device identity:

- `app_generated_device_id`

Rules:

- Generated by app
- Persisted locally
- Sent to backend for activation/license checks
- Must not depend only on hardware identifiers
- Secondary device signals may support debugging/fraud review but must not replace the primary ID

## Required Client → Platform Flows

### 1. App Version Check

Purpose:

Determine whether the client version is supported.

Client sends:

- platform
- app version
- build number
- app generated device ID if available

Platform returns:

- supported / unsupported
- minimum supported version
- force update boolean
- release notes
- update URL if applicable

### 2. Remote Config

Purpose:

Allow safe feature flags and app behavior configuration.

Client sends:

- platform
- app version
- device ID if available

Platform returns:

- maintenance mode
- enabled features
- disabled features
- safe UI/config values

Forbidden:

- stream URLs
- provider credentials
- content catalog
- hidden playlist source

### 3. Device Activation

Purpose:

Bind a user account/license to a client device.

Possible models:

- activation code
- QR pairing
- login and bind

Client sends:

- app generated device ID
- platform
- device name
- activation code or authenticated session

Platform returns:

- activation status
- device ID
- license status summary
- expiration if applicable

### 4. License Check

Purpose:

Backend-authoritative permission check for platform access.

Client sends:

- app generated device ID
- user/session token
- platform
- app version

Platform returns:

- active boolean
- reason
- expiration
- allowed features
- plan label if safe

Forbidden:

- frontend deciding license truth
- client-side fake subscription authority

### 5. Temporary Profile Transfer

Purpose:

Let a user send their own encrypted profile data to their own device temporarily.

Rules:

- User-owned
- Expiring
- Protected
- Not a marketplace
- Not a provider library
- Not public
- Not permanent default backend storage

Platform may store:

- encrypted payload
- expiration timestamp
- owner user ID
- target device ID
- consumed timestamp

Platform must not become:

- permanent provider credential authority
- public profile source
- shared playlist library

## Local-First Profile Rule

Default source/profile storage belongs on the user device.

If credentials or playlist URLs are stored, they should be stored locally and securely.

Backend storage is not the default permanent source of truth.

## Forbidden Client Behavior

Clients must not ship with:

- pirated channel lists
- default unauthorized streams
- hidden stream sources
- bundled provider credentials
- public playlist search
- provider marketplace connection
- copyrighted media files

## Final Contract Rule

A client may play media only from user-provided lawful sources or clearly authorized test/development sources that are not shipped as production content.
