# Project Intake

## Purpose

This document defines how a future project enters NEXORA Studio OS.

No product project starts until it has a clear intake record and owner approval.

Project intake must stay lean.

Do not create unnecessary departments or documents before the project is real.

## Required Project Intake Fields

Every project must define:

- name
- type
- purpose
- owner goal
- target users
- repository plan
- hosting needs
- data needs
- risk level
- first milestone
- success definition

## Project Types

Supported types:

- WEB_DESIGN
- WEB_APP
- MOBILE_APP
- TV_APP
- GAME
- BACKEND_PLATFORM
- INTERNAL_TOOL

## Lean Intake Rule

Default intake review is Director-only.

Specialist review is required only when the project has real complexity or risk.

Use Product Architect when:

- scope is unclear
- architecture is important
- product structure is complex
- long-term scalability matters

Use Builder when:

- implementation shape is unclear
- repository or stack choice is risky
- database or deployment work is needed

Use QA Reviewer when:

- security, compliance, data, or reliability risk exists
- the project may affect money, privacy, or production systems

Use Memory Writer only after owner-approved intake output needs compact storage.

## Required Foundation Records Per Project

A project should have only the records it actually needs:

- project constitution or compact brief
- roadmap or milestone list
- first task list
- decision log when decisions exist
- data rules when data exists
- risk notes when risk exists

Do not create a large document set for a small or draft project.

## Project Start Rule

A project may start only after:

- Studio Director reviews the intake
- required specialist review is completed only if justified
- owner approves the project start
- repository boundary is clear
- first milestone is clear

## Minimal Project Workflow

1. Create compact intake.
2. Director classifies risk and complexity.
3. Use specialist review only if justified.
4. Create first milestone.
5. Create first task set.
6. Approve implementation target.
7. Start work.

## Product Activation Lock

Product projects remain locked until the Studio OS runtime proves the core loop.

A product project should not begin while runtime validation or deployment gates are unresolved unless the owner explicitly overrides the lock.

## Final Rule

A project without scope, milestone, and approval stays in draft.

A project does not need a bureaucracy costume to be real.
